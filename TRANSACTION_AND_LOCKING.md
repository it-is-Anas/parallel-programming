# سلامة المعاملات والأقفال التشاؤمية (Transaction Integrity & Pessimistic Locking)

يحتوي هذا الملف على شرح هندسي وتطبيقي لكيفية ضمان **سلامة المعاملات (Transaction Integrity / ACID)** و**التحكم في الأقفال التشاؤمية (Pessimistic Concurrency Control / Locking)** في المشروع، وكيفية حماية العمليات المركبة وتحديثات المخزون الحساسة من التعارض حتى تحت ظروف الوصول المتزامن العالي، بالإضافة إلى كيفية تشغيل اختبارات الفحص.

## أولاً: سلامة المعاملات (Transaction Integrity / ACID)

### 1. المفهوم الهندسي
في تطبيقات التجارة الإلكترونية، تتكون عملية الشراء (Checkout) من عدة خطوات مركبة:
1. التحقق من توفر المنتجات في المخزون.
2. خصم الكميات المطلوبة من المخزون.
3. معالجة عملية الدفع الإلكتروني عبر بوابة الدفع.
4. إتمام الطلب وإفراغ سلة المشتريات.

من أجل الحفاظ على اتساق البيانات وتطبيق خاصية **الذرية (Atomicity)** (الكل أو لا شيء)، يجب أن تنجح جميع هذه الخطوات معاً ليتم اعتماد العملية (**Commit**)، أو إذا فشلت أي خطوة (مثل فشل الدفع)، يجب التراجع عن كل التغييرات السابقة وإعادة المخزون لحالته الأصلية (**Rollback**).

### 2. آلية التطبيق في الكود
تم تطبيق منطق ACID في الملف [orders.service.ts](file:///c:/Users/Zaid/parallel-programming/src/orders/orders.service.ts) داخل دالة الشراء `checkout` من خلال الآلية التالية:

* **أخذ لقطة حالة (Snapshot):** قبل إجراء أي تعديل على المخزون، نقوم بحفظ الكمية الأصلية لكل منتج في خريطة مؤقتة (`stockSnapshot`).
* **تنفيذ التحديثات المؤقتة:** نقوم بخصم المخزون والتحقق من الكميات.
* **محاكاة بوابة الدفع:** يتم استدعاء بوابة الدفع وفي حال حدوث فشل (يرمى استثناء/خطأ).
* **التراجع عند الفشل (Rollback):** يتم التقاط الخطأ في كتلة `catch` ويقوم النظام باستعادة حالة المخزون الأصلية المحفوظة في الـ `stockSnapshot` بشكل حتمي لكل منتج متأثر.
* **الاعتماد عند النجاح (Commit):** في حال نجاح الدفع، يتم تفريغ السلة وتثبيت التعديلات.

#### شيفرة الكود البرمجية للـ ACID:
```typescript
const stockSnapshot = new Map<string, number>();

try {
  for (const item of sortedCart) {
    // 1. حفظ الحالة الأصلية للمخزون قبل التعديل
    const product = this.db.products.get(item.productId);
    if (product) {
      stockSnapshot.set(item.productId, product.stock);
    }
  }

  // 2. خصم الكميات مؤقتاً
  for (const item of cart) {
    const product = this.db.products.get(item.productId);
    if (product) product.stock -= item.quantity;
  }

  // 3. معالجة الدفع الإلكتروني
  if (simulatePaymentFailure) {
    throw new BadRequestException('Payment failed!');
  }

  // 4. تثبيت العملية وإفراغ السلة (Commit)
  this.db.carts.set(userId, []);
} catch (error) {
  // 5. التراجع الشامل (Rollback) في حال فشل الدفع
  for (const [productId, oldStock] of stockSnapshot.entries()) {
    const product = this.db.products.get(productId);
    if (product) {
      product.stock = oldStock; // استعادة المخزون الأصلي
    }
  }
  throw error;
}
```

## ثانياً: التحكم بالأقفال التشاؤمية (Pessimistic Concurrency Control / Locking)

### 1. المفهوم الهندسي
عندما يحاول مئات المستخدمين شراء نفس السلعة ذات المخزون المحدود (مثل تذاكر مباراة أو منتج مخفض) في نفس اللحظة المتناهية، تحدث مشكلة **Race Condition** حيث يقرأ خادمان مختلفان نفس كمية المخزون ويسمحان بالبيع، مما يؤدي للبيع بالسالب.
يقوم **القفل التشاؤمي (Pessimistic Locking)** بحظر المورد بالكامل (Mutex Lock) بمجرد بدء المعاملة لطلبٍ ما، مما يجبر بقية الطلبات المتزامنة على الانتظار في طابور متسلسل (Queue) حتى يتم تحرير القفل بالكامل.

### 2. آلية التطبيق في الكود
تم استخدام مكتبة `async-mutex` عبر وحدة التحكم بالتزامن [concurrency.service.ts](file:///c:/Users/Zaid/parallel-programming/src/concurrency/concurrency.service.ts) لإنشاء قفل `Mutex` متفرد لكل منتج:

* **حجز القفل (`acquire`):** قبل قراءة أو تعديل مخزون المنتج الحساس، يقوم الطلب بحجز القفل الخاص بالمنتج.
* **منع المأزق (Deadlock Prevention):** لتجنب المأزق عندما تحاول معاملتان حجز نفس المنتجين بترتيب متعاكس، نقوم بترتيب المنتجات في السلة أبجدياً بحسب معرفها الفريد `productId` قبل البدء بحجز الأقفال.
* **المعالجة المتسلسلة والتحرير (`release`):** يتم معالجة الطلبات بالتسلسل، وبمجرد الانتهاء (سواء نجاحاً أو فشلاً في كتلة `finally`)، يتم تحرير القفل لتمكين الطلب التالي من الدخول.

#### شيفرة الكود البرمجية للأقفال:
```typescript
// ترتيب المنتجات لمنع حدوث الـ Deadlock
const sortedCart = [...cart].sort((a, b) =>
  a.productId.localeCompare(b.productId),
);
const releases: Array<() => void> = [];

try {
  // حجز الأقفال بشكل متسلسل
  for (const item of sortedCart) {
    const mutex = this.concurrencyService.getProductMutex(item.productId);
    const release = await mutex.acquire(); 
    releases.push(release);
  }

  // التحقق والتعديل الآمن داخل القفل
  // ...
} finally {
  // فك الأقفال بالترتيب المعاكس لتمكين بقية الطلبات في الطابور من العمل
  for (const release of releases.reverse()) {
    release();
  }
}
```

## ثالثاً: كيفية تشغيل الفحوصات والاختبارات (How to Run Tests)

تم إعداد ملفات اختبار منفصلة لمحاكاة الطلبات المتزامنة باستخدام استدعاءات متوازية للـ API.

### 1. تشغيل خادم الويب الرئيسي (NestJS Server)
يجب تشغيل السيرفر أولاً وترك النافذة مفتوحة:
```bash
npm run start
```

### 2. تشغيل فحص سلامة المعاملات (ACID Test)
افتح نافذة ترمينال جديدة وشغّل الأمر التالي:
```bash
node test-acid.js
```
#### النتائج المتوقعة:
1. **في وضع ACID (مع الـ Rollback):** ستلاحظ محاكاة فشل الدفع، وسيقوم السيرفر بالتراجع التلقائي وإبقاء كمية المخزون كما هي (10 قطع) لمنع استقطاع المخزون دون دفع.
2. **في وضع Non-ACID (بدون تراجع):** عند فشل الدفع، ستلاحظ بقاء المخزون ناقصاً (9 قطع) بالرغم من فشل الطلب، وهو ما يثبت حدوث عدم الاتساق عند غياب المعاملات.

### 3. تشغيل فحص الأقفال التشاؤمية (Locking Test)
في الترمينال الثاني، شغّل الأمر التالي:
```bash
node test-locking.js
```
#### النتائج المتوقعة:
1. **اختبار القفل المتفائل (Optimistic):** يرسل 5 طلبات متوازية بنفس الإصدار (Version 1). ينجح طلب واحد فقط، وتفشل الطلبات الأربعة الأخرى بخطأ `ConflictException 409` بسبب تغير رقم الإصدار في قاعدة البيانات.
2. **اختبار القفل التشاؤمي (Pessimistic):** يرسل 5 طلبات متزامنة لشراء منتج مخزونه (1). يتم حظر الطلبات ووضعها في طابور المعالجة لتسير واحداً تلو الآخر؛ فينجح الطلب الأول ويفرغ المخزون، وتفشل الطلبات الأربعة التالية بشكل منظم بخطأ `BadRequestException 400` (بسبب عدم كفاية المخزون) بدلاً من حدوث تعارض بالنسخ أو بيع بالسالب.
